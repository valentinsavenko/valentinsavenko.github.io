The Puppet community is all about automation and self-explanatory, modern approaches to infrastucture and configuration but if you decide to actually go thought all steps that are needed to have a well documented, tested and published puppet module, you'll need to read quite a while, from different sources. I try to abbreviate that process and collect _most_ relevant steps here on one page.

## 1. Create a module with the Puppet Development Kit

> Based on official docu: [installation](https://puppet.com/docs/pdk/1.x/install_pdk_osx.html) ,[usage](https://puppet.com/docs/pdk/1.x/pdk_creating_modules.html)

I assume you use a Mac, if not changes may apply and check the source material above.
<!--more-->
First let's install the Puppet Dev kit and create the module:
```bash
# install pdk
brew cask install puppetlabs/puppet/pdk

# (optional) I had to do this to add pdk to the PATH, although it's not mentioned in the install guide
echo 'export PATH=$PATH:/opt/puppetlabs/pdk/bin/' >> ~/.bash_profile

# create your module
pdk new module
```
The PDK is going to ask you about the parameters of your module like name, author, version, supported operation systems etc. but this can easily be changed later on in the `metadata.json`. This `metadata.json` contains the information than is then later _published_ to the Puppetforge and displays (hopefully helpful) descriptions for users of your module. 

After the PDK has finished a folder will be created with the module name you choose. Go into that directory and create a new class with:
```bash
pdk new class <new_module_name>
```

If the class name is the same as the module itself it will adhere to the Puppet module convention and create a init.pp, if it's a different name it will create a `.pp` file for that.
Additionally it will generate a `spec` file that's used to test that particular class at `spec/classes/<new_class_name>_spec.rb`.

This post is not about how to actually write a puppet module, but we need a module with dependecies to learn how to extend the test systems. 
So I'm going to use a very simplistic example Solr module I wrote from [here](https://github.com/valentinsavenko/puppet-solr/blob/master/manifests/init.pp) to pull examples from. You can use it as reference!

To download the dependecies that the PDK defined in the `Gemfile` we need to use the bundler and to keep it tidy and easy to change put them into `.bundle/gems`:
```bash
bundle install --path .bundle/gems
```
As I'm writing this there is a problem with the `Gemfile.lock` that is generated by the PDK (see [here](https://tickets.puppetlabs.com/si/jira.issueviews:issue-html/PDK-1213/PDK-1213.html)), to solve this I found it's enough to delete the autogenerated `Gemfile.lock` beforehand and let it be generated by the bundler during the install.


## 2. Add code and dependencies
The PDk already created a `hiera.yaml` file that's alreadsy loading the `data/commons.yaml` into the Puppet build process lets extend that with some example parameters:

```yaml
---

# Solr version to download an install
solr::version: '7.1.0'

# Sets the user for file ownership. Valid options: any valid user. Default: `solr`
solr::user: 'solr'
# Sets whether or not this instance will manage the defined user. Valid options: `true` or `false`. Default: `true`.
solr::manage_user: true

# Sets the group for file ownership. Valid options: any valid group. Default: 'solr'
solr::group: 'solr'
# Sets whether or not this instance will manage the defined group. Valid options: `true` or `false`. Default: `true`.
solr::manage_group: true

# The Solr package will be downloaded & installed here
solr::install_dir: '/opt'
```
My module is called solr, so the `init.pp` contains a class with the same name:
```ruby
class solr (
  String  $user,
  Boolean $manage_user,
  String  $group,
  Boolean $manage_group,
  String  $install_dir,
  String  $version
) {

  #------------------------------------------------------------------------------#
  # Code                                                                         #
  #------------------------------------------------------------------------------#
  # So far based on https://lucene.apache.org/solr/guide/7_1/taking-solr-to-production.html#taking-solr-to-production

  if $manage_group {
    group { $group:
      ensure => 'present',
    }
  }

  if $manage_user {
    user { $user:
      ensure => 'present',
      gid    => $group,
    }
  }

  # Download the installer archive and extract the install script
  $install_archive = "${install_dir}/solr-${$version}.tgz"
  archive { $install_archive:
    checksum_type => 'sha1',
    checksum_url  => "http://archive.apache.org/dist/lucene/solr/${$version}/solr-${$version}.tgz.sha1",
    cleanup       => false,
    creates       => 'dummy_value', # extract every time. This is needed because archive has unexpected behaviour without it. (seems to be mandatory, instead of optional)
    extract       => true,
    extract_path  => $install_dir,
    source        => "http://archive.apache.org/dist/lucene/solr/${$version}/solr-${$version}.tgz",
  }
}
```
When somebody includes your module the `init.pp` is loaded by Puppet, so this is the mandatory starting point for each module. This looks simple enough, but if somebody would load it now it would fail, because we did not specify which dependecies are needed.
We need to adjust the `metadata.json` and change the dependency section to this:
```json
  "dependencies": [
    {
      "name": "puppetlabs-stdlib",
      "version_requirement": ">= 1.0.0"
    },
    {
      "name": "puppet-archive",
      "version_requirement": ">= 3.0.0"
    }
  ],
```
This makes it clear to Puppet which packages are needed for this module and the user can download those, based on the error messages. Additionally this meta-information in displayed on the Puppetforge site later on, so that the dependency-list is known from the start.

From here on we could theoretically just use this code and be done with it, but the fun starts with getting this tested and deployed automatically.

## 3. Testing
> Sources: [Rspec](http://rspec.info/documentation/), [Rspec for Puppet](https://rspec-puppet.com/tutorial/)

I found the Rspec docu to be quite confusing if applied to Puppet, so I'll go into more detail in this section.
Puppet has no build-in, default package manager alá NPM for NodeJS, or PIP for Python. Usually what you need to do is setup Librarian Puppet, or R10K (probably better choice now, because it's used by Puppet Enterprise) on your systems and use a `Puppetfile` to define exactly which dependencies your project and (for R10K) even the dependencies of the modules you use. Only when all dependencies are available locally, can you run Puppet.
The same is the case for Rspec tests. Only that a module doesn't have a Puppetfile. We need to update the system that will download those before setting up the Puppet system to compile the puppet-catalog. For this there is a `.fixtures.yaml` and we need to make sure that it's making all needed modules available for the test run:
```yaml
---

fixtures:
  # make your own module available for the test runs
  symlinks:
    solr: "#{source_dir}"
  # dependencies from git
  repositories:
    apt: 'git://github.com/puppetlabs/puppetlabs-apt.git'
  # dependencies from puppetforge
  forge_modules:
    stdlib:
      repo: 'puppetlabs/stdlib'
      ref: '4.12.0'
    archive:  
      repo: 'puppet/archive'
      ref: '3.2.1'

```
This file will define which files will be downloaded, or symlinked to the `spec/fixtures/modules` directory. This directory is used as the default modules dir during test runs.
As you see I symlink the module name `solr` to use the current code base we're developing, change this to your module name!

The next step is writing some tests into the autogenerated test file for your class `spec/classes/solr_spec.rb`:
```ruby
require 'spec_helper'

describe 'solr' do
  context 'user, group created and Class, config' do
    let :params do
      {
        version: '7.0.0',
        install_dir: '/tmp',
        user: 'solr_special_user_name',
        group: 'solr_special_group_name',
      }
    end

    it { is_expected.to contain_user('solr_special_user_name') }
    it { is_expected.to contain_group('solr_special_group_name') }

    it { is_expected.to contain_class('Solr') }
  end

  context 'unmanaged user and group untouched' do
    let :params do
      {
        user: 'solr_special_user_name',
        group: 'solr_special_group_name',
        manage_user: false,
        manage_group: false,
      }
    end

    it { is_expected.not_to contain_user('solr_special_user_name') }
    it { is_expected.not_to contain_group('solr_special_group_name') }
  end
end
```
This simplistic test just defines which class you're testing `describe 'solr' do` and sets two contexts as example. Each context can be seen as a separate test case. In the `let :params` section you define which parameters should deviate from the defaults and the `is_expected.(not_)to` cases are checking if all behaves as you hoped with those parameters.

Important to understand is that we're not actually testing any changes to a system, as you would if you load a Virtual Machine (e.g. Vagrant, AWS ec2-instance, Docker) and let Puppet run with `puppet agent -t`. What we are testing is what change would be applied, if a system tried to run puppet. In a classical Puppet setup you have a Puppet server, or master that compiles the Puppet code and all known systems facts from the agent node and puts it together as a catalog of changes that need to be applied. This catalog is send to the agent to be applied.

We are checking whether all resources that should be existent in that catalog, are actually there. So the usecase here is more of a regression test, not a functional test.
If somebody makes updates to the code that break some old code, you'll notice it earlier. But this does not validate your changes on a system level!


Enough talk, let's run some tests:
```bash
bundle exec rake spec
```
This will download all dependencies and run all tests. 
There will be a long test output with passed and failed tests but at the end there will be something similar to this, exact number of test may vary:
```bash
Finished in 2.68 seconds (files took 6.03 seconds to load)
14 examples, 0 failures
```

## 4. Syntax and Linters
We want to use a linter for the syntax checks of Puppet- and test-code, as well as for the `metadata.json`.
Additionally the tool Rubocop can check for coding style mismatches and is configured by the generated `.rubocop` file.
To test whether your code passes a those criteria so far run one by one and fix errors of:
```bash 
bundle exec rake metadata_lint

bundle exec rake syntax lint

bundle exec rake rubocop
```
After our code is now tested, well formated and error free we can go to publish it!

## 5. CI and deployment to Puppetforge

Create your Puppetforge account and make sure that the account info is put into the `metadata.json`.
It's a good idea to update the Readme at this point and make sure it's well formated and provides good info on how to use the module.
The Readme will be the landing page of your Puppetforge module e.g. https://forge.puppet.com/valentinsavenko/solr
Now we're ready to create the package:
```bash
pdk build
```
If you didn't fill all the info already the pdk is going to ask you to add some descriptions about your module in the command line and add them to the meta data.
After it has run you'll find a compressed tar-package in the `pkg`-folder that you now can upload to the Puppetforge. 
Go to https://forge.puppet.com/upload and upload your module throught the web interface and your first module is published!

> Useful links: [Puppetforge - Travis](https://docs.travis-ci.com/user/deployment/puppetforge/), [Encrypting Travis Keys](https://docs.travis-ci.com/user/encryption-keys)

This is all fun and dandy, but we actually wanted to have a continuous delivery of our code instead of hard manual labor.
For this we adjust the `.travis.yml` with the following: 
```yaml
---
sudo: false
dist: trusty
language: ruby
cache: bundler
before_install:
  - bundle -v
  - rm Gemfile.lock || true
  - gem update --system
  - gem update bundler
  - gem --version
  - bundle -v
script:
  - 'bundle exec rake $CHECK'
bundler_args: --without system_tests
rvm:
  - 2.4.1
env:
  - PUPPET_GEM_VERSION="~> 5.0"
matrix:
  fast_finish: true
  include:
    -
      env: CHECK=rubocop
    -
      env: CHECK="syntax lint"
    -
      env: CHECK=metadata_lint
    -
      env: 
        - CHECK=spec
        - DEPLOY_TO_FORGE=true
    -
      env: PUPPET_GEM_VERSION="~> 4.0" CHECK=spec
      rvm: 2.1.9
branches:
  only:
    - master
    - /^v\d/
notifications:
  email: false
deploy:
  provider: puppetforge
  user: <puppetforge-user>
  password:
    secure: <very-long-encrypted-secret-string>
  on:
    branch: master
    condition: "$DEPLOY_TO_FORGE = true"
```
The `before_install` section sets up the ruby system environment and the `script` defines which command to run: `bundle exec rake $CHECK`. 
`matrix` shows how often that command is run and what's set for `$CHECK`, also the last test runs `spec`-tests with Puppet 4.0, instead of the usual 5.0, to make sure it's compatible.

The more interesting section is `deploy`, here we set Puppetforge as provider and need to put in our user and encrypted password.
Sign up with your Github login at https://travis-ci.com and choose the repo of your module (that's on Github and public of course ;)).
For the password we need to setup the Travis gem locally, to be able to encrypt it:
```bash
gem install travis

# will ask you to log in into travis
travis login --pro

# you need to be in the folder of the git repo for which that secret is meant, for it to be accessable by travis
travis encrypt --com '<your-pw-here>'
```
The last step encrypts your pw, prints it and pushes the public decryption key to travis, so that only the Webservice is able to decrypt it. Copy your encrypted key to the `.travis.yml` now.

After you change the version number in your `metadata.json` you're ready to deploy your module once more! Push your code changes to git and based on your `.travis.yaml`and the fact that you're registered now, Travis will start building and deploying your module in it's dashboard e.g.: https://travis-ci.com/valentinsavenko/puppet-solr

In the step where we defined the param `DEPLOY_TO_FORGE=true` you should see the deployment happening, after the test phase:
```bash
..

$ rvm $(travis_internal_ruby) --fuzzy do ruby -S gem install dpl
..
Installing deploy dependencies
..
Preparing deploy
..
Deploying application
Uploading to Puppet Forge valentinsavenko/solr
```
